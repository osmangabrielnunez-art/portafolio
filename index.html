Descripción del proyecto

Objetivo, alcance, requisitos y justificación del sistema

Objetivo general
Desarrollar un plan de proyecto unificado para el diseño de un sistema biométrico de control de asistencia
Problema que resuelve
Eliminar errores, pérdidas de información y fraudes del registro manual de asistencia
Requisitos funcionales
  • Registro de empleados dentro de la base de datos del sistema
  • Registro de asistencia mediante dispositivo biométrico de huella digital
  • Almacenamiento automático de hora de entrada y salida de cada empleado
  • Consulta del historial de asistencia por empleado y rango de fechas
  • Generación de reportes de asistencia para el área administrativa y RRHH
  • Acceso al panel de administración mediante usuario y contraseña
Requisitos no funcionales
  • Seguridad de la información almacenada en la base de datos
  • Interfaz intuitiva y fácil de usar para administradores y usuarios finales
  • Base de datos confiable con acceso rápido a los registros
  • Operatividad correcta dentro de la red interna de la empresa
Límites del proyecto
  • Propuesta estructurada — no incluye el desarrollo completo del software
  • Adquisición del hardware biométrico presentada solo como estimación presupuestal
  • Sin integración con sistemas de nómina u otras plataformas empresariales
  • El proyecto tiene fines académicos y demostrativos

Plan de calidad

Objetivos, criterios de aceptación, roles y control de versiones

Objetivos de calidad
  • Tasa de reconocimiento de huella dactilar superior al 98%
  • Cero duplicados ni registros incorrectos en la base de datos
  • Cero errores manuales que afecten reportes generados para RRHH
  • Cumplimiento del 100% de requerimientos funcionales y no funcionales
  • Seguridad de datos: acceso protegido con usuario y contraseña
  • Calificación mínima de 4/5 en pruebas de usabilidad con usuarios finales

Control de versiones — Flujo GitFlow
1

Repositorio central en GitHub con acceso controlado para cada miembro, usando ramas separadas por rol y módulo.

2

Rama main protegida — solo el Jefe de Proyecto puede aprobar y ejecutar fusiones a esta rama.

3

Ramas de desarrollo por módulo: feature/modulo-asistencia, feature/modulo-reportes, etc.

4

Commits con convención: tipo(scope): descripción — ejemplo: fix(lector): corregir fallo de reconocimiento en baja luz

5

Pull Requests obligatorios con revisión de al menos un compañero antes de fusionar cambios a main.

6

Evaluación de impacto obligatoria antes de cambios que afecten la base de datos o la interfaz de usuario.

7

Bitácora de cambios documentada con fecha, autor, descripción y justificación de cada modificación.

Cuestionario — Calidad y configuración

Respuestas fundamentadas con ejemplos del proyecto biométrico

Pregunta 1
¿Qué es un plan de calidad y por qué es importante en un proyecto de software?
Es un documento formal que define los estándares, procedimientos, recursos y responsabilidades para garantizar que el software cumpla los requisitos establecidos. Es importante porque establece criterios objetivos de éxito y evita improvisaciones costosas. En el proyecto, el plan define que el lector debe reconocer huellas con al menos 98% de precisión — sin ese criterio no habría forma objetiva de decidir si el sistema está listo para implementarse.
Pregunta 2
¿Cuál es la diferencia entre control de calidad y aseguramiento de calidad?
El aseguramiento de calidad (QA) verifica que los procesos correctos se sigan para prevenir defectos. El control de calidad (QC) inspecciona el producto para detectar y corregir defectos existentes. En el proyecto: QA sería verificar que el equipo haga pruebas unitarias después de cada módulo; QC sería ejecutar las pruebas del lector con empleados reales y medir la tasa de reconocimiento para detectar fallos concretos.
Pregunta 3
¿Qué son los KPIs y cómo se aplican en la gestión de calidad?
Son métricas cuantificables para evaluar el desempeño del proyecto respecto a sus objetivos. Permiten decisiones basadas en datos reales. Ejemplo: el KPI "tasa de errores = (errores / total pruebas) × 100" tiene meta de menos del 5%. Si al ejecutar 50 pruebas se encuentran 8 errores, la tasa es 16%, lo que activa automáticamente una revisión del módulo afectado antes de continuar.
Pregunta 4
¿Qué es la gestión de configuración y qué herramientas se utilizan?
Es el proceso de identificar, registrar y controlar versiones de todos los componentes del sistema (código, base de datos, documentación). Mantiene la integridad del sistema y facilita el trabajo colaborativo. Herramientas: Git (control distribuido), GitHub/GitLab (repositorios remotos). En el proyecto se usa Git + GitHub con ramas separadas por módulo para controlar todos los cambios del código y la documentación.
Pregunta 5
¿Cómo se controlan los cambios en un proyecto de software?
Proceso formal de 6 pasos: (1) Identificación del cambio necesario, (2) Análisis del impacto en tiempo, costo y funcionalidad, (3) Aprobación del Jefe de Proyecto, (4) Implementación en rama separada de Git, (5) Revisión por pares mediante Pull Request, (6) Documentación en bitácora con fecha, autor y justificación.
Pregunta 6
¿Qué riesgos de calidad pueden presentarse en un proyecto biométrico?
Los principales son: (a) baja tasa de reconocimiento por suciedad o características físicas del empleado; (b) pérdida de datos por fallos de hardware; (c) vulnerabilidades de seguridad que permitan acceso no autorizado; (d) resistencia de usuarios que lleve al abandono del sistema; (e) incompatibilidad técnica entre el lector adquirido y el software desarrollado.
Pregunta 7
¿Por qué es importante el control de versiones en proyectos de software?
Por cuatro razones: (1) permite recuperar versiones anteriores cuando se introduce un error grave; (2) facilita el trabajo colaborativo sin sobreescribir cambios de otros; (3) provee historial trazable de quién cambió qué y cuándo; (4) permite versiones paralelas (desarrollo, pruebas, producción) de forma ordenada. Sin Git, coordinar al analista, diseñador y equipo de apoyo generaría conflictos constantes y pérdidas de código.
